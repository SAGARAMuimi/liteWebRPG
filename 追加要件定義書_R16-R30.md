# 追加要件定義書（v1.0）— R-16〜R-30

## 1. 目的

既存の liteWebRPG（R-01〜R-15 実装済み）に対して、

- **情報可視性の向上**（何が起きているかをプレイヤーが把握しやすくする）
- **UX の改善**（操作の補助、画面設計の整理）
- **運営・コミュニティ機能の基盤整備**

を実現するための追加要件を定義する。

---

## 2. 前提・スコープ

- 対象: R-01〜R-15 が実装済みの状態を前提とする。
- UI は既存の Streamlit 方針を維持し、レイアウト幅は `layout="wide"` を継続する。
- デプロイ先は **Streamlit Community Cloud + Neon PostgreSQL** を前提とする。
- R-19 オートマッピングは R-14（マップ移動）の完成を前提とする。
- R-29 実績・称号は R-15（メタ進行）の `meta_titles` 基盤を利用する。
- R-30 ランキングはユーザー数が一定数を超えてから着手を推奨する。

---

## 3. 評価軸の定義

- 改善レベル: S（体験が大きく変わる）/ A（体験が明確に向上）/ B（局所改善）
- 開発コスト: 高（2週間以上）/ 中（3日〜2週間）/ 低（1〜2日）
- 優先度: P1（高）/ P2（中）/ P3（低）

## 5. 詳細要件

### 5.1 R-16 お知らせ機能

- 機能要件
  - `app.py` のトップページ最上部に `st.info()` でお知らせを1件表示する。
  - お知らせがない場合は何も表示しない。
  - お知らせの管理方法は以下2案から選択する。
    | 方法 | メリット | デメリット |
    |---|---|---|
    | **A: 環境変数**（`NOTICE_TEXT`） | コード変更なしで更新可（Streamlit Cloud の Secrets から設定） | 複数件・有効期限管理が難しい |
    | **B: DB テーブル**（`notices`） | 有効期限・重要度を管理できる | テーブル追加コストあり |
  - 推奨: **まず方法 A** で実装し、運用上の要求が増えたら方法 B に移行する。
  - お知らせ種別に応じてアイコン色を変える（情報 = `st.info` / 警告 = `st.warning`）。
- 受け入れ基準
  - 環境変数 `NOTICE_TEXT` が空またはなければ、お知らせブロック自体が表示されないこと。
  - 文字列が設定されていれば、ログイン前・後のいずれでも表示されること。
- 変更ファイル
  - `text_rpg/app.py`（トップページへの追記）
  - `text_rpg/config.py`（`NOTICE_TEXT` デフォルト定数追加）

### 5.2 R-17 探索ログ改善

- 機能要件
  - 探索画面（`2_dungeon.py`）の構成を以下のように再整理する。
    ```
    [① 現在地サマリー]  ← 常時表示（フロア・部屋番号・HP残量・所持アイテム数）
    [② 直前のイベント結果]  ← 最新1件のみテキスト表示
    [③ 行動ボタン群]  ← 変更なし
    ──────────────────────────────
    [④ 探索ログ ▼]  ← st.expander（デフォルト折りたたみ）
        入室 → 戦闘へ → 罠発動 → ... （全履歴）
    ```
  - `st.session_state["dungeon_log"]` の既存リストをそのまま ④ で表示する。
  - ログは新しいものが上になるよう逆順で表示する。
  - ① のサマリーには以下を1行にまとめる。
    `フロア {n}  部屋 {current_room}/{total}  ❤️{min_hp}/{min_max_hp}  🎒{item_count}個`
- 非機能要件
  - ② の直前イベント結果は Streamlit の自然な再レンダリングで更新されるため、追加の `st.rerun` は不要。
- 受け入れ基準
  - ページ上部に常時サマリーが1行表示されること。
  - 探索ログ欄が折りたたみ状態でデフォルト表示され、展開すると全履歴が確認できること。
- 変更ファイル
  - `text_rpg/pages/2_dungeon.py`

### 5.3 R-18 戦闘ログ改善

- 機能要理
  - 戦闘画面（`3_battle.py`）の構成を以下のように再整理する。
    ```
    [① 戦闘サマリー]  ← 常時表示（ターン数・生存数・バフ中フラグ）
    [② 直前ターンのログ]  ← 最新ターン分のみ st.markdown で表示
    [③ 敵ステータス]  ← 変更なし
    [④ パーティステータス]  ← R-20 で強化
    [⑤ 行動選択]  ← 変更なし
    ──────────────────────────────
    [⑥ 戦闘ログ ▼]  ← st.expander（デフォルト折りたたみ）
        ターン1: ○○が攻撃... / ターン2: ... （全履歴）
    ```
  - `st.session_state["battle_log"]` の既存リストを ⑥ で表示する。
  - ② には `battle_log` の末尾から「前回の `do_enemy_turn()` 以降」のエントリを抽出して表示する（ターン区切りを識別できるようログに区切り文字列を挿入する）。
  - ① のサマリーは `ターン {n}  生存 {alive}/{total}  敵残り {alive_enemies}体` の形式とする。
- 受け入れ基準
  - 直前ターンの結果が常に画面上部に表示されること。
  - 戦闘ログ欄が折りたたみでデフォルト表示され、展開で全ターン履歴が確認できること。
  - ターン区切りが視覚的に区別できること（例: `── ターン 3 ──` のセパレータ行）。
- 変更ファイル
  - `text_rpg/pages/3_battle.py`
  - `text_rpg/game/battle.py`（ターン区切り文字列の挿入）

### 5.4 R-19 オートマッピング

- 機能要件
  - **前提**: R-14（マップ移動・グリッドダンジョン）の実装完了後に着手する。
  - `st.session_state["visited_cells"]` に踏破済み座標 `(x, y)` のセットを保持する。
  - マップ表示は等幅フォントの ASCII グリッドで行う（例: `■` 通路 / `□` 未踏 / `@` 現在地 / `★` イベント済み）。
  - プレイ中に `st.toggle("🗺️ マップ表示")` で ON/OFF 切り替え可能とする。
  - ダンジョン終了後も `visited_cells` を `dungeon_progress` テーブルに JSON で保存し、次回入場時に参照できるようにする。
  - 踏破率（踏破マス数 / 全マス数）をサマリーに表示する。
- 制約
  - マップデータのサイズは1フロア最大 20×20 以内とする（ASCII 描画の可読性を確保するため）。
  - `visited_cells` の JSON カラムは `dungeon_progress.map_data` として追加する。
- 受け入れ基準
  - トグル OFF 時はマップが完全に非表示になること。
  - 探索を中断して再入場しても、踏破済みマスが再現されること。
- 変更ファイル
  - `text_rpg/models/dungeon.py`（`map_data` カラム追加）
  - `text_rpg/game/map_manager.py`（ASCII 描画ロジック）
  - `text_rpg/pages/2_dungeon.py`（マップ表示トグル追加）
  - `text_rpg/models/database.py`（マイグレーション）

### 5.5 R-20 ステータス表示改善

- 機能要件
  - 戦闘画面（`3_battle.py`）のパーティカードに ATK / DEF を1行追加する（最小実装・即時着手可）。
  - カードに `st.expander("詳細 ▼")` を追加し、展開時に以下を表示する（中規模実装）。
    | 表示項目 | 値の種類 |
    |---|---|
    | Lv / EXP（次Lvまで） | `chara.level` / `chara.exp` |
    | ATK / DEF | 基礎値（装備込み） |
    | INT（知性） | `chara.intelligence` |
    | 装備スロット（武器/防具/アクセサリ） | 装備名 or「なし」 |
  - 探索画面（`2_dungeon.py`）のパーティ欄にも同様の展開表示を追加する。
  - 戦闘中は expander を折りたたみ状態でデフォルト表示し、画面を圧迫しないようにする。
- 受け入れ基準
  - 戦闘画面のパーティカードに ATK / DEF が常時表示されること。
  - 展開すると Lv / EXP / 装備スロットが確認できること。
  - 展開・折りたたみ操作でページが再レンダリングされても戦闘状態（ターン・バフ等）がリセットされないこと。
- 変更ファイル
  - `text_rpg/pages/3_battle.py`（パーティカード拡充）
  - `text_rpg/pages/2_dungeon.py`（パーティカード拡充）

### 5.7 R-22 フロアクリア統計

- 機能要件
  - 戦闘勝利後の結果画面（`3_battle.py` の `is_party_victorious()` ブロック）に統計サマリーを追加する。
  - `battle_log` リストから以下を集計して表示する。
    | 統計項目 | 集計方法 |
    |---|---|
    | 総与ダメージ | ログ内の「Xダメージ」数値を正規表現で合算 |
    | 総被ダメージ | 同上（対象がパーティメンバーの行） |
    | 使用 MP 合計 | ログ内の「MP -X」を合算 or `battle_log` に専用エントリを追加 |
    | 経過ターン数 | `st.session_state["battle_turn"]` をそのまま使用 |
  - `st.metric` を使って横並びで表示する。
- 受け入れ基準
  - 勝利後の画面に4項目が横並びで表示されること。
  - ログのフォーマット変更（R-18 のターン区切り追加）と矛盾しないこと。
- 変更ファイル
  - `text_rpg/pages/3_battle.py`
  - `text_rpg/game/battle.py`（MP 消費ログを統一フォーマットで記録する場合）

### 5.8 R-23 ダンジョン入場前チェック

- 機能要件
  - ダンジョン選択画面（`2_dungeon.py` の `render_dungeon_select()`）で、各ダンジョンカードに推奨レベルと現パーティとの比較結果を表示する。
  - 判定ロジック（`config.py` の `DIFFICULTY_PRESETS` と `FLOOR_MAPS` の情報を活用）。
    | 条件 | 表示 |
    |---|---|
    | パーティ平均 Lv ≥ 推奨 Lv | `✅ 推奨レベル到達済み` |
    | パーティ平均 Lv が推奨 Lv -2 以内 | `⚠️ やや厳しいかもしれません` |
    | パーティ平均 Lv が推奨 Lv -3 以下 | `🔴 難易度が高い可能性があります` |
  - 難易度選択プルダウンの変更時にもリアルタイムで判定を更新する。
- 受け入れ基準
  - ダンジョン選択カードに判定結果が1行表示されること。
  - パーティが未設定の場合は表示しないこと（`party` が空の場合）。
- 変更ファイル
  - `text_rpg/pages/2_dungeon.py`
  - `text_rpg/config.py`（各ダンジョンの推奨レベル定数追加）

### 5.9 R-24 サイドバー装備・バフ早見表

- 機能要件
  - 探索中（`2_dungeon.py`）・戦闘中（`3_battle.py`）の `st.sidebar` に、現パーティの以下を常時表示する。

- 受け入れ基準
  - サイドバーにパーティ情報が表示されること。
  - バフが付いた場合・解除された場合にサイドバーの表示が更新されること（通常の Streamlit 再レンダリングで自動更新）。
- 変更ファイル

### 5.10 R-25 敵図鑑

### 5.12 R-27 ゲーム内チュートリアル

- 機能要件
  - `users` テーブルに `tutorial_done INTEGER DEFAULT 0` カラムを追加する。
  - 初回ログイン後、キャラクター管理画面（`1_character.py`）にステップ式の案内バナーを表示する。
    | ステップ | 内容 | 完了トリガー |
    |---|---|---|
    | 1 | キャラクターを作成しよう | キャラ作成完了 |
    | 2 | パーティを3人に編成しよう | パーティ保存完了 |
    | 3 | ダンジョンに入ろう | `2_dungeon.py` への遷移 |
    | 4 | 最初の戦闘を乗り越えよう | 1回勝利 |
  - `st.info` + 「次へ」ボタン形式で案内し、全ステップ完了または「スキップ」で `tutorial_done=1` にする。
  - `tutorial_done=1` のユーザーにはバナーを表示しない。
- 受け入れ基準
  - 新規登録ユーザーにのみチュートリアルが表示されること。
  - スキップまたは全完了後は二度と表示されないこと。
- 変更ファイル

### 5.13 R-28 バトルスピード設定

- 機能要件
  - `users` テーブルに `battle_speed VARCHAR(8) DEFAULT 'normal'` カラムを追加する。
    | キー | `time.sleep` 値 | 説明 |
    |---|---|---|
    | `none` | 0 秒 | テキスト送りなし |
    | `fast` | 0.3 秒 | 速め |
    | `normal` | 0.8 秒 | 標準（現在のデフォルト） |
  - `1_character.py` または `app.py` のサイドバーに `st.radio` で選択 UI を追加し、変更時に DB へ保存する。
  - 戦闘中の `time.sleep()` 呼び出し箇所（`3_battle.py`）で、セッションから設定値を読み取って使用する。
  - ローカル開発中のデバッグ時は `none` を推奨とするコメントをコードに残す。
- 受け入れ基準
  - 設定変更が次の戦闘から即座に反映されること。
  - `none` 設定時に `time.sleep(0)` が呼ばれても動作に影響がないこと。
- 変更ファイル

### 5.14 R-29 実績・称号システム

- 機能要件
  - **前提**: R-15（メタ進行）の `meta_titles` 基盤を使用する。
  - `config.py` に `ACHIEVEMENT_CONDITIONS` 辞書を追加し、称号名・解除条件・説明を定義する。
    | 称号 ID | 称号名 | 解除条件 |
    |---|---|---|
    | `first_win` | 初陣突破 | 初回戦闘勝利 |
    | `perseverance` | 不屈の冒険者 | 累計全滅 1 回（R-15 で既存） |
    | `explorer_10` | 探索者 | フロア 10 踏破 |
    | `no_damage` | 無傷の勝者 | 被ダメ0でフロアクリア |
    | `all_class` | 万能術師 | 8 クラス全てでパーティ編成経験あり |
  - `user.meta_titles` (JSON) に解除済み称号 ID を追記する処理を `utils/helpers.py` の `check_and_grant_achievements()` として実装する。
  - `1_character.py` の「メタ進行」セクションに取得済み称号の一覧を表示する。
  - 新たに称号を取得した際は `st.balloons()` + `st.success()` で演出する。
- 受け入れ基準
  - 各条件を満たした際に対応する称号が付与されること。
  - 同一称号の重複付与が起きないこと。
  - 全滅時に付与される `perseverance` は R-15 の既存処理と重複しないよう統合すること。
- 変更ファイル
  - `text_rpg/config.py`（`ACHIEVEMENT_CONDITIONS` 追加）
  - `text_rpg/utils/helpers.py`（`check_and_grant_achievements()` 追加）
  - `text_rpg/pages/3_battle.py`（勝利・全滅時の称号チェック呼び出し）
  - `text_rpg/pages/1_character.py`（称号一覧表示）

## 9. リスクと対策

| リスク | 対策 |
|---|---|
| R-17/R-18 の `st.expander` 内でボタンを押すと Streamlit が全体を再レンダリングし戦闘状態がリセットされる | expander 内にボタンを置かない（ログは読み取り専用）。行動ボタンは expander の外に配置する |
| R-20 の `st.expander` 展開時にDB アクセス（装備情報取得）が追加発生し遅延が増える | 装備情報は戦闘開始時に `st.session_state["party_equipment"]` にキャッシュし、expander 内ではセッションから読む |
| R-27 のチュートリアルを途中で離脱した場合に状態が中途半端になる | ステップ番号を DB に保存し、ログイン時に途中から再開できるようにする |
| R-29 の `check_and_grant_achievements()` が複数箇所から呼ばれて重複付与が起きる | 付与前に `meta_titles` に含まれていないかチェックするガードを関数内に実装する |
| R-30 の集計クエリが重くなる | `dungeon_results` に `(dungeon_id, difficulty, turns)` の複合インデックスを追加する |
